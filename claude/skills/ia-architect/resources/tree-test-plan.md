# ツリーテスト計画（Findability検証）

## 1. 目的
- 階層/ラベルだけで、ユーザーが目的情報へ辿れるか評価する
  参考: https://www.nngroup.com/articles/tree-testing/

### ツリーテストを使うタイミング
- **リデザインプロジェクトの始め**: 現行IAのベンチマーク、問題領域の発見
- **カードソートの後**: カードソートは曖昧な結果を出すため、得られた階層を必ず検証
- **コンテンツ/レイアウト作成前**: デザイン・コーディング不要で複数IA案をテスト可能

### 定性的 vs 定量的（目的に応じて選択）
| 種類 | 目的 | 人数 | 形態 | 特徴 |
|---|---|---:|---|---|
| **定性的** | 初期段階でアイデアを高速に検証・反復 | 5〜8 | モデレート推奨 | 洞察重視、統計には不適切 |
| **定量的** | ベンチマーク、複数案の比較 | 15〜50+ | 非モデレート可 | 成功率・時間を正確に測定 |

**重要**: 定量的テストを行う前に、必ず小規模な**パイロットテスト**（定性的）を実施すること。
- タスク文が理解可能か確認
- 「なぜその回答を選んだか」の洞察を得る
- ラベルの混乱ポイントを事前発見

### 複数ツリー（A/Bテスト）の設計
**ラベル比較**（同カテゴリに別名を検討）: 複数ツリーを用意し、between-subjects設計（各参加者は1ツリーのみ見る）

**場所比較**（同リソースを別カテゴリに配置検討）: 1ツリーで可能。どちらのカテゴリをクリックしたか、どちらを先に試したかを計測。

### ツールの制約（注意）
多くのツリーテストツールは**leaf（末端）のみを正解に指定可能**で、branch（中間カテゴリ）を正解にできない。
実際のサイトでは中間カテゴリにもランディングページがある場合があるため、分析時に「正しい親カテゴリを選んだか」も手動で確認する。

## 3. タスク設計（5〜15）
各タスクに「正解ノード」と「成功条件」を必ず定義。
タスクはツリー自体と同じくらい重要。以下のタスクタイプを組み合わせて設計する。

### タスクタイプ（必ず意識して設計）
| タイプ | 目的 | 例 |
|---|---|---|
| **Warmup** | テスト手順に慣れる + 不真面目な参加者のスクリーニング | 明らかに簡単なタスク（間違えたら注意散漫の証拠） |
| **Resource-finding** | 重要なビジネス目標（主要製品・サービス・情報）への到達確認 | 「○○を申し込むページを探して」 |
| **Sleight-of-hand** | 親カテゴリの「情報の匂い（information scent）」を評価 | 重要度は低いが、カテゴリ名の妥当性を測れるリソース（例: 福利厚生の細かい制度） |
| **Problem-area** | カードソートで意見が割れた箇所、新設カテゴリ、懸念点の検証 | 「このカテゴリ名で迷わず辿れるか？」 |
| **Label/Location比較** | 同一リソースに対する代替ラベル・代替配置の優劣を比較 | A/Bテスト的にツリーを分岐させる |

### タスク文のフレージング（必ず守る）
タスク文はカテゴリラベルをテストするもの。**答えを与えない・長すぎない**の2点を守る。

| NG/OK | 例 | 問題点 |
|---|---|---|
| NG | 「起業に関する情報を見つけてください」 | ラベル（Starting a Business）をそのまま使い、答えを明かしている |
| NG | 「来年サンタフェに引っ越す予定で、到着後は芝生管理サービスを提供する副業を始めて収入を補いたいと考えています。守るべき規制を調べましょう。」 | 長すぎて余計な情報が多く、参加者の注意が逸れる |
| OK | 「芝生ケアサービスの開設を検討しています。プロセスを始めるのに役立つリソースを探してください。」 | シナリオで動機を伝えつつ、簡潔でラベルを使っていない |

**フレージングチェック:**
- [ ] カテゴリ/ページのラベル語をそのまま使っていない（プライミング防止）
- [ ] シナリオ・動機で文脈を与えている
- [ ] 1〜2文で簡潔（長文に埋もれて重要部分を見落とさせない）

## 4. 参加者
- 属性:
- 人数:
  - 定性的: 5〜8名
  - 定量的: 15〜50名以上
- 実施形態:
  - モデレート（定性的推奨）
  - 非モデレート（定量的で使用可）

## 5. 指標
- **成功率**: 正解ノードに到達した割合
- **直接成功率（directness）**: 迷わず一直線に正解へ到達した割合（ラベルの明確さ・ユーザーの自信を示す）
- **時間**: タスク完了までの時間（中央値）
- **誤分岐**: どのカテゴリで迷子になったか
- **最初に試したカテゴリ**: 場所比較時に重要

## 6. 非モデレートテストの補完策
非モデレートではフォローアップ質問ができないため、以下で補う:

### テスト後アンケート（推奨）
- 混乱したラベルのリストを提示し、理解しにくかったものにチェックしてもらう
- オープンエンド質問: 「探しにくかった点、気づいたことがあれば教えてください」

### パイロットセッション（必須）
定量テスト前に数名でモデレート付きパイロットを実施。
- タスク文の理解度を確認
- 定量データでは見えないニュアンスを捉える
