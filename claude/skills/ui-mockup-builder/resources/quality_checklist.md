# Quality Checklist

モックアップ作成後、以下の項目をチェックし、問題があれば修正する。

---

## 1. Fidelity(フィデリティの適切さ)

- [ ] 目的に対してフィデリティが適切(早期なら作り込みすぎない)
- [ ] ワイヤーフレームで済む段階なのに高フィデリティにしていない
- [ ] 合意形成に必要な情報が揃っている
- [ ] 不要な装飾で議論が色・好みに寄っていない

---

## 2. States(状態の網羅性)

- [ ] 最低限の状態が揃っている(通常/空/ローディング/エラー/成功)
- [ ] 空状態(0件)でユーザーが何をすべきか明確
- [ ] ローディング状態でSkeleton UIまたはSpinnerが定義されている
- [ ] エラー状態でメッセージと回復手段が明確
- [ ] 成功状態でフィードバック(Toast等)が定義されている
- [ ] 無効状態(disabled)で理由がツールチップ等で説明されている
- [ ] 静的画で誤解されやすい挙動が注釈で補足されている

---

## 3. Content(コンテンツの現実性)

- [ ] ダミーテキスト(Lorem ipsum)を使っていない
- [ ] 実データに近いサンプルで桁数・改行・可変長を検証している
- [ ] 長いテキストが表示崩れしないか確認している
- [ ] i18n(多言語)を考慮し、英語・日本語で文字数が変わる場合を想定している
- [ ] 画像サイズ・アスペクト比が現実的
- [ ] 数値・日付のフォーマットが統一されている

---

## 4. Components & Consistency(コンポーネントと一貫性)

- [ ] 新規コンポーネントの仕様が `components/` に定義されている
- [ ] 同じ要素(ボタン、カード等)が画面間でズレていない
- [ ] デザイントークン(色/余白/タイポ)に寄せている
- [ ] 既存のデザインシステムがあれば、それに準拠している
- [ ] 都度描き(ad-hoc)でコンポーネントを作っていない
- [ ] variants / states / props が明確に定義されている

---

## 5. Layout & Spacing(レイアウトと余白)

- [ ] グリッドシステムが定義されている
- [ ] 余白スケールが統一されている(4px/8px/16px等)
- [ ] コンテナ幅が定義されている
- [ ] 主要セクションの領域分割が明確
- [ ] レスポンシブ対応の主要ブレークポイントが定義されている

---

## 6. Responsive(レスポンシブ対応)

- [ ] 主要ブレークポイント(768px/1024px)で何が変わるか定義されている
- [ ] モバイルでの表示崩れがない
- [ ] タブレットでの中途半端なレイアウトがない
- [ ] 画像・テーブルがモバイルでどう表示されるか明確
- [ ] ナビゲーション・サイドバーの折りたたみ方が定義されている

---

## 7. Accessibility(アクセシビリティ)

- [ ] 色のコントラストがWCAG AA準拠(4.5:1以上)
- [ ] フォーカス表示が明確(フォーカスリング)
- [ ] キーボード操作が可能(Tab/Enter/Space/Esc)
- [ ] スクリーンリーダー対応の注意点が記載されている(aria-label等)
- [ ] エラー通知がスクリーンリーダーで読み上げられる(aria-live)
- [ ] 画像にalt属性が付与される前提
- [ ] 最小タップ領域が44x44px以上(モバイル)

---

## 8. Interaction & Behavior(インタラクションと挙動)

- [ ] ホバー時の挙動が定義されている
- [ ] クリック時の挙動が定義されている
- [ ] フォーカス時の挙動が定義されている
- [ ] バリデーションのタイミング(リアルタイム/フォーカスアウト/送信時)が明確
- [ ] 非同期処理中のローディング状態が定義されている
- [ ] 二重送信防止の考慮がされている
- [ ] アニメーション・トランジションの方針が定義されている

---

## 9. Handoff(開発引き渡し)

- [ ] READMEに対象画面一覧と到達条件が記載されている
- [ ] 各画面の仕様ファイルへのリンクが整理されている
- [ ] 未確定事項(Open Questions)が明確で、誰がいつ決めるか記載されている
- [ ] 暫定仮定(Assumptions)が明記されている
- [ ] 重要なデザイン判断の理由が記載されている
- [ ] 関連ドキュメント(PRD/画面遷移図/デザインシステム)へのリンクがある

---

## 10. Documentation(ドキュメント品質)

- [ ] 各ファイルにMeta情報(Last updated, Owner, Status)がある
- [ ] セクションが論理的に整理されている
- [ ] 表・箇条書きで読みやすく整理されている
- [ ] コード例がシンタックスハイライト付きで記載されている
- [ ] スクリーンショット・図がある場合、パスが正しい
- [ ] リンク切れがない

---

## 11. Scope(スコープの適切さ)

- [ ] 作成する画面数が目的に対して適切
- [ ] 重要な画面を優先して詳細化している
- [ ] 全画面を同じフィデリティで作ろうとして時間を浪費していない
- [ ] MVP範囲に集中している
- [ ] 未来の要件を先取りして複雑にしていない

---

## Self-Review Questions(自己レビュー質問)

以下の質問に「はい」と答えられるか確認する。

1. **目的達成**:このモックアップで、当初の目的(合意形成/開発引き渡し)が達成できるか?
2. **誤解防止**:静的画だけでは誤解されやすい挙動が注釈で補足されているか?
3. **実装可能性**:開発者がこの仕様を見て、迷わず実装に入れるか?
4. **再利用性**:コンポーネントが画面間で再利用できる設計になっているか?
5. **スケーラビリティ**:今後画面が増えても、このスタイルガイド・トークンで対応できるか?
6. **メンテナンス性**:半年後に見ても、何をどう決めたか分かるか?

---

## Common Pitfalls(よくある落とし穴)

避けるべき典型的な問題:

- ❌ 空状態を考慮せず、データがある前提だけで設計
- ❌ エラー状態を「後で考える」として放置
- ❌ ダミーテキストで画面を埋め、実データで崩れる
- ❌ プライマリボタンを複数並べて、どれが主要か不明
- ❌ コントラストが低く、視認性が悪い
- ❌ モバイルでのレイアウトを考慮せず、デスクトップだけで設計
- ❌ コンポーネントを都度描きで、画面間でズレが発生
- ❌ フォーカス表示がなく、キーボード操作ができない
- ❌ 注釈がなく、静的画だけで挙動が誤解される
- ❌ 開発引き渡し用の情報が不足し、実装時に質問攻めになる
